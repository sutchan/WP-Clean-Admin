# 文档管理工具使用指南\r\n\r\n## 1. 概述\r\n\r\n本指南旨在介绍 WP Clean Admin 项目中使用的文档管理工具，包括如何使用这些工具来创建、编辑、管理、版本控制和发布文档。通过遵循本指南，团队成员可以高效地协作管理项目文档，确保文档的质量、一致性和可访问性。\r\n\r\n## 2. 文档管理工具栈\r\n\r\nWP Clean Admin 项目使用以下文档管理工具：\r\n\r\n| 工具名称 | 用途 | 推荐版本 |\r\n|----------|------|----------|\r\n| Markdown | 文档编写格式 | 最新规范 |\r\n| VS Code | 代码和文档编辑器 | 最新版本 |\r\n| Git | 文档版本控制 | 2.30+ |\r\n| GitHub | 文档托管和协作平台 | 最新版本 |\r\n| GitHub Actions | 文档自动化构建和部署 | 最新版本 |\r\n| Markdownlint | Markdown 格式检查 | 最新版本 |\r\n| Pandoc | 文档格式转换 | 2.19+ |\r\n\r\n## 3. Markdown 编写指南\r\n\r\n### 3.1 Markdown 基础\r\n\r\nMarkdown 是一种轻量级标记语言，易于阅读和编写，同时可以转换为 HTML、PDF 等多种格式。WP Clean Admin 项目的所有文档都使用 Markdown 格式编写。\r\n\r\n### 3.2 Markdown 语法规范\r\n\r\n#### 3.2.1 标题\r\n\r\n使用 # 符号表示标题，# 数量表示标题级别（1-6级）：\r\n\r\n```markdown\r\n# 一级标题\r\n## 二级标题\r\n### 三级标题\r\n#### 四级标题\r\n##### 五级标题\r\n###### 六级标题\r\n```\r\n\r\n#### 3.2.2 段落和换行\r\n\r\n- 段落之间使用空行分隔\r\n- 行尾添加两个空格表示换行\r\n\r\n#### 3.2.3 列表\r\n\r\n- **无序列表**：使用 - 或 * 符号\r\n- **有序列表**：使用数字加 . 符号\r\n\r\n```markdown\r\n- 无序列表项 1\r\n- 无序列表项 2\r\n  - 嵌套列表项\r\n\r\n1. 有序列表项 1\r\n2. 有序列表项 2\r\n   1. 嵌套有序列表项\r\n```\r\n\r\n#### 3.2.4 链接和图片\r\n\r\n- **链接**：`[链接文本](链接地址 "可选标题")`\r\n- **图片**：`![图片描述](图片地址 "可选标题")`\r\n\r\n```markdown\r\n[GitHub](https://github.com/sutchan/WPCleanAdmin)\r\n![WP Clean Admin Logo](https://example.com/logo.png)\r\n```\r\n\r\n#### 3.2.5 强调\r\n\r\n- **粗体**：`**粗体文本**` 或 `__粗体文本__`\r\n- **斜体**：`*斜体文本*` 或 `_斜体文本_`\r\n- **删除线**：`~~删除线文本~~`\r\n\r\n#### 3.2.6 代码\r\n\r\n- **行内代码**：使用反引号 `` ` `` 包裹\r\n- **代码块**：使用三个反引号 `` ``` `` 包裹，并指定语言\r\n\r\n```markdown\r\n行内代码：`wp plugin activate wp-clean-admin`\r\n\r\n代码块：\r\n```bash\r\n# 克隆代码\r\ngit clone https://github.com/sutchan/WPCleanAdmin.git\r\n```\r\n```php\r\n// PHP 代码示例\r\nfunction wpca_cleanup_menus() {\r\n    // 清理菜单逻辑\r\n}\r\n```\r\n```json\r\n{\r\n  "name": "wp-clean-admin",\r\n  "version": "1.7.15"\r\n}\r\n```\r\n```dotenv\r\n# 环境变量配置\r\nDB_NAME=wp_clean_admin\r\nDB_USER=root\r\n```\r\n```yaml\r\n# YAML 配置示例\r\ntheme:\r\n  name: material\r\n  features:\r\n    - navigation.tabs\r\n```\r\n```sql\r\n-- SQL 查询示例\r\nSELECT * FROM wp_options WHERE option_name LIKE 'wpca_%';\r\n```\r\n```javascript\r\n// JavaScript 代码示例\r\nfunction wpca_init() {\r\n  console.log('WP Clean Admin initialized');\r\n}\r\n```\r\n```css\r\n/* CSS 代码示例 */\r\n.wpca-cleaned-menu {\r\n  display: none;\r\n}\r\n```\r\n```html\r\n<!-- HTML 代码示例 -->\r\n<div class="wpca-settings">\r\n  <h2>WP Clean Admin 设置</h2>\r\n</div>\r\n```\r\n\r\n#### 3.2.7 表格\r\n\r\n使用 | 分隔列，使用 - 分隔表头和表体：\r\n\r\n```markdown\r\n| 列 1 | 列 2 | 列 3 |\r\n|------|------|------|\r\n| 内容 1 | 内容 2 | 内容 3 |\r\n| 内容 4 | 内容 5 | 内容 6 |\r\n```\r\n\r\n#### 3.2.8 引用\r\n\r\n使用 > 符号表示引用：\r\n\r\n```markdown\r\n> 这是一段引用文本。\r\n> \r\n> 引用可以包含多行。\r\n```\r\n\r\n#### 3.2.9 分隔线\r\n\r\n使用三个或更多的 -、* 或 _ 符号表示分隔线：\r\n\r\n```markdown\r\n---\r\n***\r\n___\r\n```\r\n\r\n#### 3.2.10 任务列表\r\n\r\n使用 - [ ] 表示未完成任务，- [x] 表示已完成任务：\r\n\r\n```markdown\r\n- [x] 完成文档编写\r\n- [ ] 审核文档\r\n- [ ] 发布文档\r\n```\r\n\r\n### 3.3 Markdown 最佳实践\r\n\r\n1. **保持简洁**：使用简洁明了的语言，避免冗长复杂的句子\r\n2. **结构清晰**：使用合适的标题层级，组织文档结构\r\n3. **格式一致**：保持文档格式的一致性，包括缩进、换行、列表样式等\r\n4. **使用代码块**：对于代码示例，使用代码块并指定语言\r\n5. **添加链接**：对于引用的外部资源，添加链接\r\n6. **使用图片**：对于复杂的概念，使用图片辅助说明\r\n7. **添加目录**：对于长文档，添加目录便于导航\r\n8. **使用语义化标记**：使用合适的 Markdown 标记表示内容语义\r\n\r\n## 4. VS Code 文档编辑指南\r\n\r\n### 4.1 VS Code 安装和配置\r\n\r\n1. **下载安装**：从 [VS Code 官网](https://code.visualstudio.com/) 下载并安装 VS Code\r\n2. **安装扩展**：安装以下推荐扩展\r\n   - **Markdown All in One**：Markdown 编辑增强\r\n   - **Markdownlint**：Markdown 格式检查\r\n   - **Prettier**：代码和文档格式化\r\n   - **GitLens**：Git 增强\r\n   - **GitHub Pull Requests and Issues**：GitHub 集成\r\n   - **Paste Image**：快速粘贴图片\r\n\r\n### 4.2 VS Code 基本使用\r\n\r\n1. **打开文档**：使用 VS Code 打开项目目录，导航到 docs 文件夹\r\n2. **创建文档**：右键点击 docs 或子文件夹，选择 "新建文件"，输入文件名（如 `new-document.md`）\r\n3. **编辑文档**：在编辑器中编写 Markdown 内容\r\n4. **预览文档**：按下 `Ctrl+Shift+V` 或点击编辑器右上角的预览按钮，查看 Markdown 渲染效果\r\n5. **格式化文档**：按下 `Shift+Alt+F` 或右键点击编辑器，选择 "格式化文档"，使用 Prettier 格式化文档\r\n6. **检查格式**：Markdownlint 会自动检查文档格式，在问题面板中显示格式错误\r\n\r\n### 4.3 VS Code 高级功能\r\n\r\n1. **多光标编辑**：按住 `Alt` 键，点击多个位置，进行多光标编辑\r\n2. **代码片段**：使用 Markdown 代码片段快速插入常用内容\r\n3. **查找替换**：按下 `Ctrl+F` 查找，`Ctrl+H` 替换\r\n4. **文件搜索**：按下 `Ctrl+Shift+F` 搜索整个项目中的内容\r\n5. **Git 集成**：使用 VS Code 内置的 Git 功能，进行提交、推送、拉取等操作\r\n\r\n## 5. Git 文档版本控制\r\n\r\n### 5.1 Git 基础\r\n\r\nGit 是一种分布式版本控制系统，用于跟踪文件的变化。WP Clean Admin 项目的文档使用 Git 进行版本控制。\r\n\r\n### 5.2 Git 工作流程\r\n\r\n1. **克隆仓库**：\r\n   ```bash\r\n   git clone https://github.com/sutchan/WPCleanAdmin.git\r\n   cd WPCleanAdmin\r\n   ```\r\n\r\n2. **创建分支**：\r\n   ```bash\r\n   # 创建并切换到新分支\r\n   git checkout -b docs/update-readme\r\n   ```\r\n\r\n3. **编辑文档**：使用 VS Code 或其他编辑器编辑文档\r\n\r\n4. **添加更改**：\r\n   ```bash\r\n   # 添加所有更改\r\n   git add .\r\n   \r\n   # 或添加特定文件\r\n   git add docs/README.md\r\n   ```\r\n\r\n5. **提交更改**：\r\n   ```bash\r\n   git commit -m "docs: 更新 README.md 文档"\r\n   ```\r\n\r\n6. **推送分支**：\r\n   ```bash\r\n   git push -u origin docs/update-readme\r\n   ```\r\n\r\n7. **创建 Pull Request**：在 GitHub 上创建 Pull Request，请求将分支合并到 main 分支\r\n\r\n8. **代码审查**：团队成员审查 Pull Request，提供反馈\r\n\r\n9. **合并分支**：审查通过后，合并分支到 main 分支\r\n\r\n10. **删除分支**：\r\n    ```bash\r\n    # 删除本地分支\r\n    git branch -d docs/update-readme\r\n    \r\n    # 删除远程分支\r\n    git push origin --delete docs/update-readme\r\n    ```\r\n\r\n### 5.3 Git 最佳实践\r\n\r\n1. **使用有意义的提交信息**：提交信息应清晰描述更改内容，遵循 Conventional Commits 规范\r\n2. **频繁提交**：每次提交应包含一个逻辑上的更改，避免一次性提交大量不相关的更改\r\n3. **使用分支**：为每个文档更新创建单独的分支，避免直接在 main 分支上编辑\r\n4. **定期拉取**：定期从远程仓库拉取最新更改，避免冲突\r\n5. **解决冲突**：如果出现冲突，及时解决冲突\r\n6. **使用标签**：为重要的文档版本创建 Git 标签\r\n\r\n### 5.4 Conventional Commits 规范\r\n\r\n提交信息应遵循以下格式：\r\n\r\n```\r\n<type>(<scope>): <description>\r\n\r\n[optional body]\r\n\r\n[optional footer(s)]\r\n```\r\n\r\n- **type**：提交类型，包括\r\n  - `feat`：新功能\r\n  - `fix`：修复 bug\r\n  - `docs`：文档更新\r\n  - `style`：代码风格更改\r\n  - `refactor`：代码重构\r\n  - `test`：测试相关\r\n  - `chore`：构建或工具相关\r\n\r\n- **scope**：提交范围，如 `readme`、`installation`、`api` 等\r\n\r\n- **description**：提交描述，简洁明了\r\n\r\n**示例**：\r\n```\r\ndocs(readme): 更新项目概述和功能特点\r\ndocs(installation): 添加 Docker 部署说明\r\ndocs(api): 更新核心函数参考文档\r\n```\r\n\r\n## 6. GitHub 文档协作\r\n\r\n### 6.1 GitHub 仓库结构\r\n\r\nWP Clean Admin 项目的文档存储在 GitHub 仓库的 `docs` 目录中，按照功能和类型组织：\r\n\r\n```\r\ndocs/\r\n├── 01-文档指南/          # 文档编写指南和规范\r\n├── 02-项目概述/          # 项目概述和介绍\r\n├── 03-使用指南/          # 插件使用指南\r\n├── 04-功能模块/          # 功能模块详细说明\r\n├── 05-技术文档/          # 技术文档和 API 参考\r\n├── 06-开发指南/          # 开发指南和规范\r\n├── 07-测试文档/          # 测试策略和计划\r\n├── 08-部署文档/          # 部署指南和方案\r\n├── 09-运维文档/          # 运维手册和指南\r\n├── 10-工具文档/          # 工具使用指南\r\n├── 11-附录/              # 附录和参考资料\r\n├── CHANGELOG.md          # 变更日志\r\n├── README.md             # 项目 README\r\n└── 项目审查报告_*.md     # 项目审查报告\r\n```\r\n\r\n### 6.2 GitHub 文档协作流程\r\n\r\n1. **创建 Issue**：对于需要更新的文档，创建 GitHub Issue，描述需要更新的内容和原因\r\n2. **分配任务**：将 Issue 分配给相应的团队成员\r\n3. **创建分支**：根据 Issue 创建相应的分支\r\n4. **编辑文档**：在分支上编辑文档\r\n5. **创建 Pull Request**：提交 Pull Request，关联相应的 Issue\r\n6. **代码审查**：团队成员审查 Pull Request，提供反馈\r\n7. **合并 Pull Request**：审查通过后，合并 Pull Request 到 main 分支\r\n8. **关闭 Issue**：合并后，关闭相应的 Issue\r\n\r\n### 6.3 GitHub 文档查看和访问\r\n\r\n1. **在线查看**：直接在 GitHub 上查看 Markdown 文档\r\n2. **克隆仓库**：克隆仓库到本地，使用 VS Code 或其他编辑器查看\r\n3. **GitHub Pages**：如果配置了 GitHub Pages，可以通过 GitHub Pages 访问构建后的文档网站\r\n\r\n## 7. Markdownlint 文档格式检查\r\n\r\n### 7.1 Markdownlint 安装和配置\r\n\r\n1. **安装 Markdownlint CLI**：\r\n   ```bash\r\n   npm install -g markdownlint-cli\r\n   ```\r\n\r\n2. **创建配置文件**：在项目根目录创建 `.markdownlint.json` 配置文件\r\n\r\n3. **配置规则**：\r\n   ```json\r\n   {\r\n     "default": true,\r\n     "MD003": {"style": "atx"},\r\n     "MD004": {"style": "dash"},\r\n     "MD007": {"indent": 2},\r\n     "MD010": false,\r\n     "MD013": false,\r\n     "MD024": {"allow_different_nesting": true},\r\n     "MD025": {"front_matter_title": "# WP Clean Admin"},\r\n     "MD033": false,\r\n     "MD034": false,\r\n     "MD041": false\r\n   }\r\n   ```\r\n\r\n### 7.2 Markdownlint 使用\r\n\r\n1. **检查单个文件**：\r\n   ```bash\r\n   markdownlint docs/README.md\r\n   ```\r\n\r\n2. **检查所有文档**：\r\n   ```bash\r\n   markdownlint docs/**/*.md\r\n   ```\r\n\r\n3. **自动修复问题**：\r\n   ```bash\r\n   markdownlint docs/**/*.md --fix\r\n   ```\r\n\r\n4. **在 VS Code 中使用**：安装 Markdownlint 扩展后，VS Code 会自动检查文档格式，并在问题面板中显示错误\r\n\r\n### 7.3 常见 Markdownlint 规则\r\n\r\n| 规则 ID | 规则名称 | 描述 |\r\n|---------|----------|------|\r\n| MD003 | Heading style | 标题样式，推荐使用 atx 样式（# 符号） |\r\n| MD004 | Unordered list style | 无序列表样式，推荐使用 dash 样式（- 符号） |\r\n| MD007 | Unordered list indentation | 无序列表缩进，推荐使用 2 个空格 |\r\n| MD010 | Hard tabs | 禁止使用硬制表符，使用空格代替 |\r\n| MD013 | Line length | 行长度限制，推荐不超过 80 个字符 |\r\n| MD024 | Multiple headings with the same content | 禁止多个相同内容的标题 |\r\n| MD025 | Multiple top-level headings in the same document | 每个文档只能有一个一级标题 |\r\n| MD033 | Inline HTML | 禁止使用内联 HTML |\r\n| MD034 | Bare URL used | 禁止使用裸露的 URL，应使用链接格式 |\r\n| MD041 | First line in file should be a top-level heading | 文件第一行应为一级标题 |\r\n\r\n## 8. Pandoc 文档格式转换\r\n\r\n### 8.1 Pandoc 安装和配置\r\n\r\n1. **下载安装**：从 [Pandoc 官网](https://pandoc.org/) 下载并安装 Pandoc\r\n2. **验证安装**：\r\n   ```bash\r\n   pandoc --version\r\n   ```\r\n\r\n### 8.2 Pandoc 基本使用\r\n\r\n1. **Markdown 转 HTML**：\r\n   ```bash\r\n   pandoc docs/README.md -o docs/README.html\r\n   ```\r\n\r\n2. **Markdown 转 PDF**：\r\n   ```bash\r\n   pandoc docs/README.md -o docs/README.pdf\r\n   ```\r\n\r\n3. **Markdown 转 Word**：\r\n   ```bash\r\n   pandoc docs/README.md -o docs/README.docx\r\n   ```\r\n\r\n4. **批量转换**：\r\n   ```bash\r\n   for file in docs/**/*.md; do\r\n     pandoc "$file" -o "${file%.md}.html"\r\n   done\r\n   ```\r\n\r\n### 8.3 Pandoc 高级使用\r\n\r\n1. **使用模板**：\r\n   ```bash\r\n   pandoc docs/README.md --template=template.html -o docs/README.html\r\n   ```\r\n\r\n2. **添加元数据**：\r\n   ```bash\r\n   pandoc docs/README.md -o docs/README.pdf --metadata title="WP Clean Admin 项目文档" --metadata author="Sut" --metadata date="2025-11-30"\r\n   ```\r\n\r\n3. **生成目录**：\r\n   ```bash\r\n   pandoc docs/README.md -o docs/README.pdf --toc\r\n   ```\r\n\r\n4. **指定 CSS**：\r\n   ```bash\r\n   pandoc docs/README.md -o docs/README.html -c style.css\r\n   ```\r\n\r\n## 9. 文档自动化构建和部署\r\n\r\n### 9.1 GitHub Actions 配置\r\n\r\n1. **创建 workflow 文件**：在项目根目录创建 `.github/workflows/docs.yml` 文件\r\n\r\n2. **配置 workflow**：\r\n   ```yaml\r\n   name: Build and Deploy Docs\r\n   \r\n   on:\r\n     push:\r\n       branches: [ main ]\r\n     pull_request:\r\n       branches: [ main ]\r\n   \r\n   jobs:\r\n     build:\r\n       runs-on: ubuntu-latest\r\n       steps:\r\n       - uses: actions/checkout@v3\r\n       \r\n       - name: Setup Node.js\r\n         uses: actions/setup-node@v3\r\n         with:\r\n           node-version: '18'\r\n       \r\n       - name: Install dependencies\r\n         run: npm install -g markdownlint-cli\r\n       \r\n       - name: Lint Markdown files\r\n         run: markdownlint docs/**/*.md\r\n       \r\n       - name: Build HTML docs\r\n         run: |\r\n           mkdir -p build/docs\r\n           for file in docs/**/*.md; do\r\n             pandoc "$file" -o "build/${file%.md}.html"\r\n           done\r\n       \r\n       - name: Deploy to GitHub Pages\r\n         if: github.ref == 'refs/heads/main'\r\n         uses: peaceiris/actions-gh-pages@v3\r\n         with:\r\n           github_token: ${{ secrets.GITHUB_TOKEN }}\r\n           publish_dir: ./build/docs\r\n   ```\r\n\r\n### 9.2 自动化流程\r\n\r\n1. **提交触发**：当代码提交到 main 分支或创建 Pull Request 时，GitHub Actions 自动触发\r\n2. **格式检查**：使用 Markdownlint 检查文档格式\r\n3. **构建文档**：使用 Pandoc 将 Markdown 文档转换为 HTML\r\n4. **部署文档**：将构建后的 HTML 文档部署到 GitHub Pages\r\n\r\n## 10. 文档管理最佳实践\r\n\r\n### 10.1 文档编写最佳实践\r\n\r\n1. **明确目标**：明确文档的目标受众和用途\r\n2. **结构清晰**：使用清晰的标题层级和目录，组织文档结构\r\n3. **内容准确**：确保文档内容准确，与实际功能一致\r\n4. **语言简洁**：使用简洁明了的语言，避免冗长复杂的句子\r\n5. **格式一致**：保持文档格式的一致性，包括缩进、换行、列表样式等\r\n6. **定期更新**：定期更新文档，确保文档与代码同步\r\n7. **添加示例**：对于复杂的概念，添加示例代码或截图\r\n8. **使用版本控制**：使用 Git 进行文档版本控制，跟踪文档变化\r\n\r\n### 10.2 文档协作最佳实践\r\n\r\n1. **分配责任**：明确每个文档的负责人\r\n2. **定期审查**：定期审查文档，确保文档的质量和准确性\r\n3. **使用 Issue 跟踪**：使用 GitHub Issue 跟踪文档更新任务\r\n4. **使用 Pull Request 审查**：所有文档更新都通过 Pull Request 进行审查\r\n5. **提供反馈**：对于文档审查，提供具体的反馈和建议\r\n6. **鼓励贡献**：鼓励团队成员参与文档编写和更新\r\n\r\n### 10.3 文档发布最佳实践\r\n\r\n1. **选择合适的发布方式**：根据目标受众选择合适的文档发布方式，如 GitHub Pages、PDF、Word 等\r\n2. **保持最新**：确保发布的文档是最新版本\r\n3. **提供导航**：为文档网站提供清晰的导航，便于用户查找内容\r\n4. **优化搜索**：为文档网站添加搜索功能，便于用户搜索内容\r\n5. **收集反馈**：提供反馈渠道，收集用户对文档的反馈\r\n\r\n## 11. 总结\r\n\r\n本指南介绍了 WP Clean Admin 项目中使用的文档管理工具，包括 Markdown 编写、VS Code 编辑、Git 版本控制、GitHub 协作、Markdownlint 格式检查、Pandoc 格式转换以及 GitHub Actions 自动化构建和部署。\r\n\r\n通过遵循本指南，团队成员可以高效地协作管理项目文档，确保文档的质量、一致性和可访问性。文档管理是项目成功的重要组成部分，良好的文档可以提高团队的协作效率，降低新成员的学习成本，提升项目的整体质量。\r\n\r\n随着项目的发展，文档管理工具和流程可能会不断优化和改进，团队成员应关注最新的文档管理实践，持续提高文档管理的效率和质量。
