# WP Clean Admin 文档格式规范

## 1. 概述

为了确保 WP Clean Admin 项目文档的一致性、可读性和可维护性，特制定本文档格式规范。所有项目文档都应遵循本规范，确保文档风格统一，便于阅读和维护。

## 2. 基本要求

### 2.1 编码格式
- 所有文档文件必须使用 **UTF-8 编码，无 BOM**
- 换行符统一使用 **Unix LF**
- 禁止使用 Windows CRLF 换行符

### 2.2 文件命名
- 文件名使用中文，清晰描述文件内容
- 文件名使用连字符 `-` 分隔单词，例如：`文档格式规范.md`
- 禁止使用空格、下划线或其他特殊字符
- 文件扩展名统一为 `.md`

### 2.3 目录结构
- 目录名使用中文，清晰描述目录内容
- 目录名格式为：`XX-目录名称`，例如：`01-文档指南`
- 目录编号使用两位数字，从 01 开始
- 禁止使用空格、下划线或其他特殊字符

## 3. Markdown 语法规范

### 3.1 标题层级

#### 3.1.1 标题层级规范
- 一级标题：`#`，用于文档标题
- 二级标题：`##`，用于主要章节
- 三级标题：`###`，用于章节下的小节
- 四级标题：`####`，用于小节下的子小节
- 五级标题：`#####`，用于子小节下的细目
- 六级标题：`######`，用于细目下的具体内容

#### 3.1.2 标题格式要求
- 标题前后各空一行
- 标题与 `#` 之间有一个空格
- 标题文字首字母大写
- 标题文字不使用标点符号

**示例：**
```markdown
# 文档标题

## 主要章节

### 小节标题

#### 子小节标题
```

### 3.2 段落

- 段落之间空一行
- 每行不超过 80 个字符（便于阅读和版本控制）
- 禁止使用连续空格或制表符缩进
- 段落首行不缩进

### 3.3 代码块

#### 3.3.1 行内代码
- 使用反引号 `` ` `` 包裹行内代码
- 代码与反引号之间无空格

**示例：**
```markdown
使用 `wpca_get_settings()` 函数获取插件设置。
```

#### 3.3.2 代码块
- 使用三个反引号 ``` 包裹代码块
- 代码块前空一行
- 代码块后空一行
- 指定代码语言，便于语法高亮
- 代码缩进使用 4 个空格
- 代码行长度不超过 80 个字符

**示例：**
```markdown
```php
// 获取插件设置
function wpca_get_settings( $key = '', $default = false ) {
    $settings = get_option( 'wpca_settings', array() );
    
    if ( empty( $key ) ) {
        return $settings;
    }
    
    return isset( $settings[$key] ) ? $settings[$key] : $default;
}
```
```

### 3.4 列表

#### 3.4.1 无序列表
- 使用 `-` 作为无序列表标记
- 列表标记与内容之间有一个空格
- 列表项之间空一行
- 嵌套列表使用 4 个空格缩进

**示例：**
```markdown
- 功能一
  - 子功能一
  - 子功能二
- 功能二
- 功能三
```

#### 3.4.2 有序列表
- 使用数字 `.` 作为有序列表标记
- 列表标记与内容之间有一个空格
- 列表项之间空一行
- 嵌套列表使用 4 个空格缩进

**示例：**
```markdown
1. 第一步
   1. 子步骤一
   2. 子步骤二
2. 第二步
3. 第三步
```

### 3.5 链接

#### 3.5.1 行内链接
- 格式：`[链接文本](链接地址 "可选标题")`
- 链接文本清晰描述链接内容
- 链接地址使用完整 URL
- 禁止使用缩写 URL

**示例：**
```markdown
[WP Clean Admin GitHub 仓库](https://github.com/sutchan/WP-Clean-Admin "WP Clean Admin GitHub 仓库")
```

#### 3.5.2 参考链接
- 适合多次引用同一链接
- 格式：`[链接文本][链接标识符]`
- 链接定义：`[链接标识符]: 链接地址 "可选标题"`
- 链接定义放在文档末尾

**示例：**
```markdown
[WP Clean Admin][1] 是一个功能强大的 WordPress 后台管理和优化插件。

[1]: https://github.com/sutchan/WP-Clean-Admin "WP Clean Admin GitHub 仓库"
```

### 3.6 图片

#### 3.6.1 图片格式
- 格式：`![图片描述](图片地址 "可选标题")`
- 图片描述清晰准确
- 图片地址使用相对路径或完整 URL
- 禁止使用缩写 URL

**示例：**
```markdown
![WP Clean Admin 插件截图](images/wpca-screenshot.png "WP Clean Admin 插件截图")
```

#### 3.6.2 图片尺寸
- 图片宽度不超过 800px
- 图片高度自适应
- 禁止使用过大的图片

### 3.7 表格

#### 3.7.1 表格格式
- 表格前后各空一行
- 表头与内容使用 `-` 分隔
- 表格对齐使用 `:` 标记
- 表格内容居中对齐（默认）
- 表格列宽自适应

**示例：**
```markdown
| 功能模块 | 描述 | 优先级 |
|----------|------|--------|
| 后台清理 | 清理 WordPress 后台冗余内容 | 高 |
| 性能优化 | 提高网站加载速度 | 高 |
| 安全增强 | 增强网站安全性 | 中 |
```

#### 3.7.2 表格对齐
- 左对齐：`| 左对齐 |`
- 居中对齐：`| :--- |`
- 右对齐：`| ---: |`

**示例：**
```markdown
| 左对齐 | 居中对齐 | 右对齐 |
| :----- | :------: | -----: |
| 内容1  | 内容2    | 内容3  |
| 内容4  | 内容5    | 内容6  |
```

### 3.8 强调

#### 3.8.1 粗体
- 使用 `**` 包裹粗体文本
- 粗体与 `**` 之间无空格
- 用于强调重要内容

**示例：**
```markdown
**重要提示：** 请确保备份数据库后再进行操作。
```

#### 3.8.2 斜体
- 使用 `*` 包裹斜体文本
- 斜体与 `*` 之间无空格
- 用于强调或引用

**示例：**
```markdown
*注意：* 此功能仅适用于 WordPress 5.0+ 版本。
```

#### 3.8.3 删除线
- 使用 `~~` 包裹删除线文本
- 删除线与 `~~` 之间无空格
- 用于表示已删除或过时的内容

**示例：**
```markdown
~~此功能已废弃~~
```

### 3.9 引用

- 使用 `>` 作为引用标记
- 引用标记与内容之间有一个空格
- 引用前后各空一行
- 嵌套引用使用多个 `>` 标记

**示例：**
```markdown
> 这是一个引用。
> 
> > 这是一个嵌套引用。
```

### 3.10 注释

- 使用 HTML 注释格式：`<!-- 注释内容 -->`
- 注释内容清晰描述注释目的
- 禁止使用过多注释
- 注释放在适当位置，不影响文档可读性

**示例：**
```markdown
<!-- 以下是功能模块列表 -->
- 功能一
- 功能二
- 功能三
```

## 4. 文档内容规范

### 4.1 内容组织

- 文档内容逻辑清晰，结构合理
- 内容由浅入深，循序渐进
- 重点内容突出显示
- 避免冗余内容

### 4.2 语言规范

- 使用简洁、准确的中文
- 避免使用口语化表达
- 避免使用模糊、歧义的词汇
- 术语使用一致
- 代码示例使用英文注释

### 4.3 代码示例

- 代码示例完整、可运行
- 代码示例包含必要的注释
- 代码示例遵循项目代码规范
- 代码示例使用真实的函数和类名
- 代码示例格式正确，缩进一致

**示例：**
```php
// 获取插件版本
function wpca_get_version() {
    $plugin_data = get_plugin_data( WPCA_PLUGIN_FILE );
    return $plugin_data['Version'];
}
```

### 4.4 术语表

- 文档中使用的术语应在文档开头或末尾定义
- 术语定义清晰准确
- 术语使用一致

**示例：**

## 术语表

| 术语 | 定义 |
|------|------|
| WP Clean Admin | 一个 WordPress 后台管理和优化插件 |
| BOM | Byte Order Mark，字节顺序标记 |
| LF | Line Feed，换行符 |
| CRLF | Carriage Return Line Feed，回车换行符 |

## 5. 文档结构规范

### 5.1 文档结构

所有文档应包含以下基本结构：

1. **标题**：文档的主标题
2. **概述**：文档的简要介绍和目的
3. **正文**：文档的主要内容，按章节组织
4. **结论**：文档的总结和结论
5. **参考资料**：文档中引用的外部资源
6. **术语表**：文档中使用的术语定义

### 5.2 文档模板

```markdown
# 文档标题

## 1. 概述

文档的简要介绍和目的。

## 2. 章节标题

章节内容。

### 2.1 小节标题

小节内容。

#### 2.1.1 子小节标题

子小节内容。

## 3. 结论

文档的总结和结论。

## 4. 参考资料

- [参考资料1](链接地址)
- [参考资料2](链接地址)

## 5. 术语表

| 术语 | 定义 |
|------|------|
| 术语1 | 定义1 |
| 术语2 | 定义2 |
```

## 6. 文档审核规范

### 6.1 审核内容

- 文档格式是否符合规范
- 文档内容是否完整、准确
- 文档语言是否简洁、准确
- 代码示例是否完整、可运行
- 术语使用是否一致

### 6.2 审核流程

1. 文档编写完成后，提交审核
2. 审核人员检查文档格式和内容
3. 审核人员提出修改意见
4. 文档编写人员修改文档
5. 再次提交审核，直到通过

### 6.3 审核标准

- 文档格式符合本规范
- 文档内容完整、准确
- 文档语言简洁、准确
- 代码示例完整、可运行
- 术语使用一致
- 文档易于理解和维护

## 7. 文档维护规范

### 7.1 文档更新

- 文档内容应及时更新，确保与项目实际情况一致
- 文档更新后，应在文档末尾添加更新日志
- 更新日志应包含更新时间、更新内容和更新人员

### 7.2 文档版本控制

- 文档应使用版本控制工具管理
- 文档版本号遵循语义化版本规范（MAJOR.MINOR.PATCH）
- 文档版本号与项目版本号保持一致

### 7.3 文档归档

- 过时的文档应归档保存
- 归档文档应标记为 "已归档"
- 归档文档应保留在项目中，便于查阅历史版本

## 8. 总结

本规范旨在确保 WP Clean Admin 项目文档的一致性、可读性和可维护性。所有项目文档都应遵循本规范，确保文档风格统一，便于阅读和维护。

通过遵循本规范，我们可以：
- 提高文档的可读性和可维护性
- 确保文档内容的一致性和准确性
- 便于团队成员协作和沟通
- 提高项目的整体质量

希望所有团队成员都能严格遵守本规范，共同维护高质量的项目文档。
